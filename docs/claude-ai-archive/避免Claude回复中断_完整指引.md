# 避免 "Claude's response was interrupted" 完整操作指引

## 核心原则

回复中断的本质是**输出空间不足**或**连接中断**。所有策略都围绕两个方向展开：一是主动控制输出规模使其不触及天花板，二是优化基础设施保障连接稳定性。

---

## 策略一：分段输出——最直接有效的防中断手段

### 1.1 为什么有效

Claude 单次回复有硬性输出上限（当前约 16,000 tokens，即约 12,000 英文单词或 6,000 汉字）。当你要求生成完整的长文档、大型代码文件、或详尽的分析报告时，输出很容易触及这个天花板。分段输出将一个大任务拆成多个小任务，每个都在安全范围内。

### 1.2 具体做法

**方法 A：在首条 Prompt 中预设分段结构**

```
请为我写一份关于 X 的完整报告，包含以下章节：
1. 背景与现状
2. 问题分析
3. 解决方案
4. 实施计划
5. 风险评估

请先只写第1-2节。写完后停下，等我确认再继续。
```

这样做的好处是 Claude 在规划阶段就知道总体结构，但只输出部分内容，保证了连贯性又避免了超长输出。

**方法 B：递进式生成**

第一轮：

```
请先给出这份报告的完整大纲（只要标题和每节 1-2 句摘要）。
```

第二轮：

```
大纲确认无误。请展开写第一节"背景与现状"的完整内容。
```

第三轮：

```
继续写第二节"问题分析"。
```

**方法 C：对代码文件的分段策略**

```
请实现 UserAuth 模块。由于代码较长，请按以下顺序分步输出：
1. 先输出类定义和构造函数
2. 等我说"继续"后输出核心方法（login、logout、refresh）
3. 最后输出辅助方法和导出
```

### 1.3 分段时机判断表

| 任务类型 | 预估输出量 | 是否需要分段 | 建议分段方式 |
|----------|-----------|-------------|------------|
| 简单问答 | < 500字 | 否 | — |
| 代码片段 | < 100行 | 否 | — |
| 中等文档 | 500-3000字 | 视情况 | 可一次输出，但加"如果超长请分段" |
| 完整报告 | > 3000字 | **是** | 按章节分段 |
| 大型代码文件 | > 200行 | **是** | 按模块/类/函数分段 |
| Artifact（React/HTML） | > 300行 | **是** | 先框架后细节 |
| 表格数据 | > 50行 | **是** | 按数据分组分批 |

### 1.4 防中断保险句

在任何可能产生长输出的 Prompt 末尾加上这句话：

```
如果回复可能很长，请在接近输出限制时主动停下，总结已完成的部分和待完成的部分，等我确认后继续。
```

这相当于给 Claude 一个"安全阀"——它会在感知到输出接近上限时主动分段，而不是被系统强制截断。

## 策略二：勤开新对话——释放输出空间

### 2.1 为什么有效

上下文窗口（200K tokens）需要同时容纳输入和输出。长对话中对话历史不断膨胀，挤压输出空间。一个 30 轮对话的历史可能已经占据 100K-150K tokens，留给 Claude 回复的空间可能只剩几千 tokens，远低于正常的 16K 上限。

### 2.2 何时必须开新对话

**信号 1：回复明显变短或变空泛。** 如果 Claude 的回答从之前的详尽转为简短概括，说明输出空间已被压缩。

**信号 2：连续出现中断。** 同一个对话中如果已经出现过一次中断，第二次中断的概率会更高（因为中断的回复本身也被记入对话历史，进一步挤压空间）。

**信号 3：对话轮次超过 15 轮。** 根据之前的研究，15 轮是一个经验临界点。

**信号 4：使用了大量工具调用。** 每次 web_search、web_fetch、代码执行的返回结果都会被记入上下文。一轮包含 5 次搜索的回复可能消耗 20K-50K tokens 的上下文空间。

### 2.3 开新对话的标准操作流程

**步骤 1：在旧对话中生成接力摘要**

```
请生成一份密集摘要，包含：
1. 当前任务的目标和背景（2-3句）
2. 已完成的工作和关键决策（列表形式）
3. 当前进度和下一步待做的事项
4. 任何重要的约束条件或偏好设置

摘要应该足够让一个全新的 Claude 会话无缝接续工作。
```

**步骤 2：开新对话，粘贴摘要**

```
以下是上一轮对话的工作摘要，请在此基础上继续：

[粘贴摘要]

当前需要完成的下一步是：[具体任务]
```

**步骤 3：如果是代码项目，附带关键文件**

不要只粘贴摘要——把当前版本的核心代码文件也一起提供，否则 Claude 只有描述没有实际内容。

### 2.4 利用 Projects 减少重开成本

如果你在 Project 中工作，项目知识库中的文件不需要每次重新粘贴。因此：

- 将稳定的参考文档（规范、指南、需求文档）放入项目知识库
- 只在对话中提供变化的内容（当前代码状态、最新进展）
- 这样新对话的"启动成本"可以从粘贴 10K tokens 降到 1-2K tokens

## 策略三：用文件创建替代聊天输出

### 3.1 为什么有效

当 Claude 通过代码执行功能将内容写入文件时，文件内容不受单次回复长度限制的同等约束。一个 `create_file` 调用可以写入远超聊天回复上限的内容。这是处理长文档和大型代码文件最可靠的方式。

### 3.2 触发文件创建的 Prompt 模板

**用于文档类任务：**

```
请将完整报告写入一个 Markdown 文件（不要在聊天中输出），然后提供文件下载。报告应包含：
- [具体要求]
```

**用于代码类任务：**

```
请直接创建代码文件（不要在聊天中展示代码），包含完整实现。
文件结构：
- src/auth.js — 认证模块
- src/utils.js — 工具函数
- tests/auth.test.js — 测试文件

创建完成后给我一个简要说明即可。
```

**用于数据分析任务：**

```
请将分析结果写入 Excel 文件，包含以下 Sheet：
1. 原始数据
2. 汇总统计
3. 图表数据

不需要在聊天中列出所有数据，只需要提供关键发现的文字总结。
```

### 3.3 对比：聊天输出 vs 文件创建

| 维度 | 聊天输出 | 文件创建 |
|------|---------|---------|
| 长度限制 | ~16K tokens（硬上限） | 远高于聊天上限 |
| 中断风险 | 高（长内容极易触发） | 低（写入过程在服务端完成） |
| 可编辑性 | 需要在对话中重新生成 | 可在本地编辑后重新上传 |
| Token 消耗 | 全部计入对话历史 | 文件内容不重复计入后续轮次 |
| 适用场景 | 短回答、讨论、解释 | 文档、代码、报表、数据 |

### 3.4 组合技：文件创建 + 聊天摘要

最佳实践是让 Claude 将详细内容写入文件，然后在聊天中只给出简短摘要：

```
请完成以下两件事：
1. 将完整的 API 设计文档写入 api-design.md 文件
2. 在聊天中只给我一段 3-5 句话的摘要，说明核心设计决策

这样我可以下载完整文档，同时快速了解要点。
```

## 策略四：优化 VPN/代理连接稳定性

### 4.1 为什么有效

Claude 使用 SSE（Server-Sent Events）流式传输回复。一个长回复的流式传输可能持续 30 秒到 3 分钟。在此期间任何网络中断都会导致回复被截断。通过代理访问时，连接链路更长，中断概率更高。

### 4.2 Clash Verge 优化配置

**选择低延迟节点：** 打开 Clash Verge → Proxies，按延迟排序，选择延迟 < 150ms 且丢包率为 0 的节点。避免使用"自动选择"——它可能在回复过程中切换节点，导致连接断开。

**锁定节点：** 手动指定一个稳定节点后，不要在 Claude 使用期间切换。如果用了节点组（如"美国节点"），进入组内手动选定具体节点，而非依赖负载均衡。

**启用 TUN 模式：** Clash Verge → Settings → TUN Mode → 开启。TUN 模式在系统层面接管网络流量，比 HTTP 代理模式更稳定，尤其对 WebSocket 和 SSE 长连接。

**调整超时参数：** 在 Clash 配置文件中增加连接保持时间：

```yaml
keep-alive-interval: 30  # 每30秒发送心跳
tcp-concurrent: true       # 启用并发TCP连接
```

**禁用 UDP：** 如果不需要 UDP 流量，在规则中禁用。UDP 相关的 DNS 查询失败偶尔会拖慢整体连接。

### 4.3 浏览器端优化

**禁用节电模式：** 笔记本电脑的节电模式会降低网络性能。在使用 Claude 时保持"高性能"电源计划。

**关闭不必要的标签页：** 大量打开的标签页会竞争网络带宽和浏览器资源，影响 SSE 流的接收稳定性。

**使用有线网络：** 如果条件允许，用网线替代 Wi-Fi。Wi-Fi 的微波炉干扰、信号波动等都可能导致瞬间断连，而 SSE 流对此非常敏感。

### 4.4 连接中断后的快速恢复

如果回复中途中断了：

**场景 A：回复已输出大部分内容**

```
你的回复在"[粘贴最后几个字]"处中断了。请从断点处继续，不要重复已输出的内容。
```

**场景 B：回复才刚开始就中断了**

直接重新发送原始 Prompt。不需要额外解释"上次中断了"——这只是浪费 token。

**场景 C：频繁中断（同一对话中连续 2 次以上）**

这通常意味着上下文已经过满。立即执行策略二（开新对话），不要再尝试在当前对话中继续。

## 策略五：控制工具调用的上下文膨胀

### 5.1 为什么有效

每次 web_search 返回约 2K-5K tokens 的结果，web_fetch 可能返回 5K-20K tokens，代码执行的 stdout/stderr 也会进入上下文。一个涉及 5 次搜索 + 3 次网页抓取的回合可能一次性消耗 50K+ tokens 的上下文空间，严重挤压后续输出空间。

### 5.2 精确控制搜索范围

**错误做法：**

```
帮我全面调研 AI Agent 的最新进展，包括学术论文、商业产品、开源项目、社区讨论等所有方面。
```

这会触发大量搜索和抓取，迅速填满上下文。

**正确做法：**

```
关于 AI Agent 最新进展，我最关心以下 2 个方面：
1. 2026 年发布的主要商业 Agent 产品
2. 最受关注的开源 Agent 框架

请针对这 2 点各做 1 次搜索，给我关键要点即可。不需要抓取完整网页。
```

### 5.3 限制 Deep Research 和 Extended Thinking 的使用

**Deep Research** 是 token 消耗最快的功能——一次深度研究可能消耗数万 tokens。只在真正需要全面调研时使用，日常问题用普通搜索即可。

**Extended Thinking** 的思考过程虽然对你不可见，但仍消耗上下文空间。对于简单任务，关闭 Extended Thinking 可以释放输出空间。只在需要复杂推理（数学证明、多步骤规划、代码架构设计）时开启。

### 5.4 文件上传的 Token 控制

**上传前预处理：**

- PDF 文件：如果只需要其中几页，先用 PDF 工具提取相关页面再上传
- 代码文件：只上传相关模块，不要上传整个项目
- 图片：预先压缩到合理分辨率（Claude 内部会将图片缩放到长边 1568px，上传更大的图片只是浪费上传时间和 token）

**上传后引导聚焦：**

```
我上传了一份 50 页的文档。请只关注第 3 章"技术架构"部分（约第 15-25 页），回答以下问题：[问题]
```

## 策略六：模型选择与输出长度控制

### 6.1 模型与中断概率的关系

| 模型 | 输出倾向 | 中断风险 | 适用场景 |
|------|---------|---------|---------|
| Opus | 极为详尽、面面俱到 | **高** | 复杂分析、架构决策、长期策略 |
| Sonnet | 适度详细、重点突出 | **中** | 日常开发、内容创作、数据分析 |
| Haiku | 简洁高效、直奔主题 | **低** | 快速问答、代码审查、文档生成 |

### 6.2 针对不同模型的防中断策略

**使用 Opus 时必须分段：**

Opus 的回复天然偏长，几乎任何中等复杂度以上的任务都应该预设分段。同时在 Prompt 中加入长度约束：

```
请用 Opus 的深度分析能力，但控制总输出在 2000 字以内。聚焦最关键的 3 个要点，每个要点 500-700 字。
```

**使用 Sonnet 时的最佳平衡：**

Sonnet 是中断风险和输出质量的最佳平衡点。大部分任务用 Sonnet，只在需要"一锤定音"的关键决策时切换 Opus。

**使用 Haiku 的安全场景：**

Haiku 几乎不会触发中断（输出天然简短），适合：
- 快速代码审查（"这段代码有什么问题？"）
- 简单格式转换（"把这个 JSON 转成 YAML"）
- 头脑风暴（"给我 10 个标题建议"）

### 6.3 输出长度的显式控制

在 Prompt 中明确要求输出长度是最可靠的防中断手段：

```
请用 300 字以内回答。
```

```
请给出 5 个要点，每个要点 1-2 句话。
```

```
请写一段不超过 500 字的摘要。
```

这些指令会让 Claude 主动压缩输出，远离截断区间。

## 策略七：中断后的高效恢复工作流

### 7.1 标准恢复流程

```
第一步：评估中断位置
  ├── 回复 > 80% 完成 → 直接请求补完尾部
  ├── 回复 30-80% 完成 → 请求从断点继续
  └── 回复 < 30% 完成 → 重新发送，加分段指令

第二步：选择恢复方式
  ├── 同一对话上下文充足 → 在当前对话继续
  └── 同一对话已很长 → 开新对话 + 粘贴已完成部分

第三步：防止再次中断
  └── 在恢复请求中加入分段指令或长度限制
```

### 7.2 恢复 Prompt 模板

**补完尾部（内容已基本完成）：**

```
你的回复在 "[最后几个字]" 处中断。请只输出剩余部分（从断点开始），不要重复已输出的内容。
```

**从断点继续（内容完成了一半）：**

```
你的回复在写到 [某个章节/某个函数] 时中断了。已完成的部分我已保存。

请从 [具体位置] 继续输出后续内容。如果剩余内容仍然很长，请分 2 次输出。
```

**完全重做（内容才刚开始）：**

```
[原始 Prompt]

注意：请将回复控制在 2000 字以内。如果需要更多篇幅，请先输出前半部分，我确认后再继续。
```

### 7.3 对于 Artifact 中断的特殊处理

Artifact（代码/HTML/React）中断特别棘手，因为不完整的代码无法运行。恢复策略：

```
上一个 Artifact 在 [描述位置] 处中断，代码不完整。请重新生成完整的 Artifact。

为避免再次中断，请：
1. 先写核心框架（状态管理 + 主要组件结构）
2. 样式和细节可以简化
3. 总代码控制在 200 行以内
```
